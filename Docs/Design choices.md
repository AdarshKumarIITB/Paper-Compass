
# MVP Design Spec: AI Research Assistant

## Product Summary

An AI-powered research assistant for semi-technical professionals ramping on academic CS topics. Three modes: Discover (find papers), Evaluate (decide if a paper is worth reading), Comprehend (depth-adjustable, visual-first explanations with threaded side conversations). The core differentiator is that explanations are visual by default, generated by AI with no human curation.

Target user: technical enough to want real details, not necessarily an active researcher in the paper's field.

---

## Part 1: Core Interaction Model

### Entry Points

Two entry points, not three. Comprehend is a transition from Evaluate, not an independent starting point. A user who already knows they want to deep-dive a paper still needs the system to parse it, which is functionally Evaluate with an immediate "go deeper" action.

**Topic entry (Discover).** User provides a topic string, optionally with scope constraints (time range, subfield). System returns a ranked set of papers with title, authors, year, citation count and a 2-sentence contribution summary.

**Paper entry (Evaluate).** User provides a URL, DOI or PDF upload. System produces a structured evaluation: what the paper claims, what method it uses, how strong the evidence is, what prerequisite knowledge is needed to read it.

Home screen surfaces both entry points. No separate Comprehend entry.

Exception: returning users who previously evaluated a paper and want to comprehend it. Handled through the library/session state, not a third entry point.

### Transitions

Directed graph with three nodes. Valid transitions:

|From|To|Trigger|
|---|---|---|
|Discover|Evaluate|User selects a paper from results|
|Evaluate|Comprehend|User commits to deep reading|
|Evaluate|Discover|User wants related work or the broader landscape|
|Comprehend|Discover|User encounters a cited concept or paper to explore|
|Comprehend|Evaluate|User clicks a reference and wants to know if it's worth reading|
|Discover|Discover|User refines query or pivots to adjacent area|

**Blocked transition for MVP:** Discover → Comprehend. The system needs the parsing step from Evaluate to generate good explanations.

**Branching, not abandoning:** transitioning from Comprehend to Discover or Evaluate preserves the comprehension session. The user is branching within a single investigation.

### Thread Model (Within Comprehend)

**Trigger:** always user-initiated. User selects a concept, term, section or visual element and asks a question or requests more detail. System can suggest threadable concepts via inline affordances on terms it detects as potentially unfamiliar, but never auto-opens a thread.

**What a thread inherits:**

- Parent paper's full context
- The specific passage or visual that spawned the thread
- User's current depth level
- Concept history from the main branch (so the thread doesn't re-explain things the user has already seen)
- User's demonstrated knowledge level (per Gray & Holyoak principle 1: analogies must map from what the reader already knows)

**What carries back to the main branch:**

- Terms explained in threads lose their "unfamiliar" affordance in the main branch
- If the thread produced a useful analogy or diagram, a collapsed reference appears inline in the main branch at the relevant point
- The thread itself remains accessible as a collapsible node

**Thread visuals:** threads get their own visuals, positioned next to thread content (Mayer spatial contiguity). Thread visuals inherit the same color coding and signaling conventions as the main branch.

**Thread depth:** inherits the global depth setting by default. User can adjust within a thread independently. Expected common pattern: main branch at intermediate depth, thread opened at lower depth to explore a prerequisite concept.

**MVP nesting constraint:** one level of thread depth. Threads cannot spawn sub-threads. If the user needs to go deeper within a thread, the thread itself expands.

### Session and State

**Returning user view:** a library showing every paper the user has touched, grouped by status (discovered, evaluated, reading, completed). Each paper card shows the last interaction point for resuming.

**No explicit "research project" for MVP.** Implicit grouping from discovery sessions: if a user ran a Discover query on "RLHF alignment techniques" and evaluated three papers from those results, those papers share a lineage. Display that lineage as a lightweight grouping in the library without requiring the user to name or manage projects.

**Per-paper persisted state:**

- Evaluation summary
- Comprehension progress (which sections expanded, which threads opened)
- Depth setting at time of last interaction
- User bookmarks and highlights

### Edge Cases

**Paper already in history.** Surface the existing evaluation with a note: "You evaluated this on [date]. Here's where you left off." Offer re-evaluate (useful if user's background has changed) or resume.

**Discovery surfaces an already-evaluated paper.** Show it in results with a badge indicating prior evaluation and the user's verdict. Don't hide it; the user may want to revisit in a new context.

**Cross-paper comparison.** Deferred to V2 (requires aligned section mapping between papers). For MVP, users open two papers in separate sessions and switch between them. The library view provides a lightweight comparison surface with both evaluation summaries visible together.

**Paper references another paper the user hasn't read.** In Comprehend, cited papers appear as interactive references. Clicking one triggers an inline micro-evaluation: 3-sentence summary of what the cited paper contributes and why the current paper cites it, without leaving the comprehension view. From there the user can open a full Evaluate session, which becomes a branch in their library.

## Part 2: Adjustable Depth Mechanism

### What Depth Controls

Five dimensions, coupled into a single axis for MVP. Independent per-dimension control deferred (combinatorial UI problem, forces decisions users can't make before reading the content).

Dimensions, in escalation order:

1. **Assumed prerequisites.** Low depth: prerequisite concepts explained inline. High depth: prerequisites assumed.
2. **Vocabulary.** Low depth: jargon introduced with definitions and analogies. High depth: standard terminology used without scaffolding.
3. **Granularity of breakdowns.** Low depth: processes broken into more steps with more narration per step. High depth: multiple steps compressed.
4. **Context and motivation.** Low depth: more space on _why_ before _what_. High depth: ratio shifts toward _what_ and _how_.
5. **Mathematical formality.** Low depth: relationships described verbally and visually. High depth: formal notation introduced.

The ordering encodes "intuition before formalism": conceptual scaffolding dimensions escalate first, mathematical formality escalates last.

### Depth in Evaluate vs. Comprehend

**Evaluate mode: fixed depth, no user adjustment.** Locked to the equivalent of the middle tier. The evaluation must be substantive enough to be informative on its own. Uses standard terminology, explains the core method with a visual, includes enough detail to assess the contribution. Letting users fiddle with depth before committing to a paper adds friction without value.

**Comprehend mode: user-selected default with per-section adjustment.** Default set during onboarding via calibration: show the user a sample explanation at intermediate depth and let them adjust up or down. This outperforms asking "what's your level?" because users are poor self-assessors. A concrete example with "is this about right?" produces a more accurate calibration.

### Three Discrete Levels

Discrete levels, not a continuous slider. A slider implies fine-grained control the system can't deliver. Three levels map to three genuinely different explanation strategies.

Labels describe explanation style, not user competence: **Conceptual / Technical / Formal** (not Beginner / Intermediate / Advanced).

**Conceptual** (default for most users):

- Precise descriptions using plain language
- Analogies with explicit A↔B mappings (Gray & Holyoak principles 1-2)
- "Where this breaks down" notes for every analogy (Gray & Holyoak principle 3)
- Visuals show process flows, system diagrams, annotated step-throughs
- Prerequisites explained inline
- No formal notation except where it's simpler than the verbal equivalent (e.g., O(n))

**Technical:**

- Standard field terminology assumed
- Analogies available but collapsed by default
- Visuals add formal annotations (variable names, dimensions, function signatures)
- Key equations presented with verbal interpretation alongside
- Prerequisites flagged but not explained (linked to thread-capable definitions)

**Formal:**

- Full mathematical treatment
- Proofs and derivations where relevant
- Visuals include computational graphs, probability graphical models
- Assumes grad-level familiarity with the subfield

### Per-Section Inline Deepening

Each section in a Comprehend session has an inline control to go one level deeper on that section only. A user globally set to Conceptual who deepens one section sees that section at Technical while the rest stays Conceptual (Mayer segmenting principle).

When the user deepens a section, visuals for that section regenerate at the new depth. The transition between visual depth levels should be animated (conceptual diagram morphs into technical one) so the user tracks which new elements correspond to existing ones (Mayer temporal contiguity).

### Intuition-Before-Formalism Expansion

When a user requests more depth on a section, the system expands in two phases within a single response:

**Phase 1 (expand intuitive explanation):** more detail on why the mechanism exists, what problem it solves, what would go wrong without it. Refine or add the analogy. Update the visual to show more internal structure while keeping the same conceptual framing.

**Phase 2 (add formal layer below):** introduce notation, equations or algorithmic pseudocode. Provide a second visual: the formal version of the diagram, annotated with variable names and operations. Include a "where this breaks down" note.

Phase 2 is visually distinct (different background, different visual style) so the user can identify and skip it at a glance. The user can stop reading after Phase 1 and still have gained something.

### Failure Mode Mitigations

**"Dumbed down" feeling at low depth.** Maintain claim precision while reducing prerequisite assumptions. Low depth means the explanation is complete at a conceptual level; precision lives in the reasoning, not the notation. "The model pays more attention to relevant words" is dumbed down. "The model computes a relevance score between every pair of positions in the input, then uses those scores to create a weighted mix of representations, so each position's output is influenced most by the positions scored as most relevant" is precise without being formal.

**"Wall of jargon" at high depth.** Apply Mayer's pre-training principle. Before any section at Technical or Formal depth, present the 3-5 key terms that section relies on with one-line definitions. Even at Formal depth, the reader should never encounter a term without prior introduction. Glossary can be collapsed for users who don't need it.

**Hard-to-visualize papers (theoretical proofs, negative results).** Shift visual strategy from spatial/process diagrams to structured text visuals: proof outlines as numbered trees, assumption-conclusion chains as directed graphs, counterexample constructions as before/after comparisons. A well-structured proof outline is a picture of logical structure. The Multimedia principle says words + pictures; not everything needs a flowchart.

**Level mismatch between user selection and actual level.** Track two signals: time spent per section (unusually fast = too basic, unusually slow = too advanced) and thread-opening rate (frequent threads on basic concepts = depth too high). On detected mismatch, surface a non-intrusive suggestion: "This section might work better at [other level]. Adjust?" Never auto-adjust. Collect signal data in V1, build auto-adjustment model in V2.

## Concrete Example: Attention Mechanism at Three Depths

### Conceptual

**Text:** "For each word in the input, the model asks: which other words are relevant to understanding this one? It scores every pair of words for relevance, then builds a new representation of each word that's a weighted blend of all the others, with the most relevant words contributing the most. The result is that each word's representation now carries information from the words that matter to it, regardless of how far apart they are in the sentence."

**Visual:** a sentence with 5 words, arrows from one highlighted word to all others with thickness proportional to relevance score, output representation shown as a colored blend.

**Analogy:** "Like scanning a crowded room when someone says a name. Your attention jumps to the faces most likely to be relevant, and you build your understanding of the situation primarily from those people, even if they're standing far apart."

**Mapping:** words = people in the room, query = the name you heard, attention scores = how much you focus on each person, output = your updated understanding.

**Where this breaks down:** unlike scanning a room, the model computes all scores simultaneously and doesn't have a single focus point.

### Technical

**Text:** "The input embeddings are projected into three spaces via learned linear transformations: Queries (Q), Keys (K) and Values (V). Attention scores are computed as the scaled dot product of Q and K, normalized via softmax. The output is the weighted sum of V using these scores. The scaling factor 1/sqrt(d_k) prevents the dot products from growing large enough to push softmax into regions with vanishingly small gradients."

**Visual:** same 5-word diagram, now with Q/K/V projection arrows, a matrix showing score computation, softmax normalization and resulting weighted combination. Dimension labels (d_model, d_k, d_v) annotated.

### Formal

**Text:** Full equation: Attention(Q, K, V) = softmax(QK^T / sqrt(d_k))V, with derivation of why sqrt(d_k) is the correct scaling factor (variance analysis of dot products in high dimensions), and discussion of multi-head attention as parallel attention functions on projected subspaces.

**Visual:** computational graph showing tensor dimensions at each operation, with multi-head parallel structure and final concatenation + projection.

## Research Principles Embedded in the Design

Reference map showing where each principle from Gray & Holyoak and Mayer surfaces in the spec:

|Principle|Where it appears|
|---|---|
|Gray & Holyoak 1 (familiar source domain)|Thread model inherits user's demonstrated knowledge; depth calibration via concrete example|
|Gray & Holyoak 2 (explicit mapping)|Conceptual depth requires A↔B mapping tables for every analogy|
|Gray & Holyoak 3 (acknowledge limitations)|Every analogy includes a "where this breaks down" note|
|Gray & Holyoak 4 (multiple analogies)|Alternative analogies offered when threads signal confusion|
|Gray & Holyoak 5 (reader inference)|Deferred to post-MVP ("check your understanding" prompts)|
|Mayer Multimedia|Visual component by default for every explanation|
|Mayer Spatial Contiguity|Visuals inline next to relevant text, threads get their own visuals|
|Mayer Temporal Contiguity|Text and visuals appear simultaneously; depth transitions animated|
|Mayer Signaling|Consistent color coding, annotated diagram parts, legend included|
|Mayer Segmenting|Per-section depth controls, expandable detail levels|
|Mayer Coherence|Every diagram element corresponds to something in the paper; no decorative elements|
|Mayer Redundancy|Short labels on diagrams, not full sentences duplicating the text|
|Mayer Pre-training|3-5 key term glossary before sections at Technical/Formal depth|
|Mayer Modality|Audio walkthroughs deferred to post-MVP|
|Mayer Personalization|Explanations generated in second person|
