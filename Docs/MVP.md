
# Product Requirements Document: AI Research Assistant (MVP)

## 1. Product Overview

An AI-powered research assistant for semi-technical professionals who need to ramp on academic CS topics. The product operates in three modes:

- **Discover**: Find relevant papers for a given topic.
- **Evaluate**: Decide whether a specific paper is worth reading.
- **Comprehend**: Read a paper with depth-adjustable, visual-first explanations and threaded side conversations.

The core differentiator: explanations are visual by default, generated by AI with no human curation. Every explanation includes a diagram or chart matched to the content type, at the reader's chosen depth level.

**Target user**: Someone technical enough to want real details but not necessarily an active researcher in the paper's field. Examples: a software engineer reading a reinforcement learning paper to evaluate a vendor's claims, a data scientist reading a systems paper outside their subfield, a technical product manager preparing for a strategy discussion on a new capability.

---

## 2. Entry Points and Navigation

### 2.1 Entry Points

Two entry points. Comprehend is a transition from Evaluate, not an independent starting point.

**Topic entry (Discover)** The user provides a topic string with optional scope constraints (time range, subfield). The system returns a ranked list of papers, each displaying: title, authors, year, citation count, influential citation count and a two-sentence contribution summary.

**Paper entry (Evaluate)** The user provides a URL, DOI or PDF upload. The system produces a structured evaluation covering: what the paper claims, what method it uses, how strong the evidence is and what prerequisite knowledge is needed to read it.

The home screen surfaces both entry points side by side. No separate Comprehend entry exists. Returning users who previously evaluated a paper access Comprehend through the library.

### 2.2 Mode Transitions

Valid transitions form a directed graph:

|From|To|Trigger|
|---|---|---|
|Discover|Evaluate|User selects a paper from results|
|Evaluate|Comprehend|User commits to deep reading|
|Evaluate|Discover|User wants related work or broader landscape|
|Comprehend|Discover|User encounters a cited concept or paper|
|Comprehend|Evaluate|User clicks a reference to assess it|
|Discover|Discover|User refines query or pivots to adjacent area|

**Blocked for MVP**: Discover directly to Comprehend. The parsing step in Evaluate is required to generate quality explanations.

**Branching, not abandoning**: Transitioning from Comprehend to Discover or Evaluate preserves the Comprehend session. The user is branching within a single investigation, not starting over.

## 3. Feature Requirements by Mode

### 3.1 Discover Mode

**Data source**: Semantic Scholar API (free authenticated access at 1 req/sec; partner key available at 100 req/sec).

**Required API endpoints**:

- `/paper/search` — semantic search by topic
- `/paper/{id}` — metadata, abstract, TLDR, citation counts, influential citation counts
- `/paper/{id}/citations` and `/paper/{id}/references` — citation graph traversal

**Discovery algorithm**:

1. Semantic search using SPECTER2 embeddings (provided by the API) to get seed papers.
2. Pull citation graphs for top results.
3. Rank by influential citation count as a proxy for foundational importance.
4. Apply recency weighting using publication date and citation velocity.

**Result display per paper**:

- Title, authors, year
- Citation count and influential citation count
- AI-generated two-sentence contribution summary (sourced from Semantic Scholar's TLDR when available, generated by the system when not)
- Badge if the user has previously evaluated or read the paper
- One-click transition to Evaluate

**User controls**:

- Topic query input (free text)
- Time range filter (optional)
- Subfield filter (optional)
- Sort order toggle: foundational importance vs. recency

### 3.2 Evaluate Mode

**Input**: URL, DOI or PDF upload.

**Output**: A structured evaluation containing:

1. **Claim summary** — What does this paper claim to contribute? (2-3 sentences)
2. **Method overview** — How does it achieve this? (3-5 sentences with one visual: architecture diagram, process flow or formulation diagram depending on the paper's content type)
3. **Evidence assessment** — How strong is the evidence? Covers: dataset size and appropriateness, baseline comparisons, ablation presence, statistical reporting, limitations acknowledged by the authors.
4. **Prerequisite map** — What does the reader need to know to read this paper? Lists required concepts with a rough familiarity indicator (e.g., "linear algebra: basic," "reinforcement learning: intermediate").
5. **Reading time estimate** — Estimated time to comprehend the paper at each depth level.

**Depth**: Fixed at the equivalent of the middle tier (Technical). No user adjustment in Evaluate mode.

**Transition affordance**: A "Read this paper" action that moves the user into Comprehend mode.

### 3.3 Comprehend Mode

**Structure**: The paper is parsed into sections. Each section is presented with:

- The paper's original text (or a close paraphrase where copyright constraints apply)
- An AI-generated explanation at the user's selected depth level
- An AI-generated visual matched to the section's content type (see Section 5)
- Inline affordances on terms the system detects as potentially unfamiliar

**Depth control**: A global depth selector (Conceptual / Technical / Formal) plus per-section inline deepening. Details in Section 4.

**Thread model**: User-initiated side conversations. Details in Section 6.

**Progress tracking**: The system tracks which sections the user has expanded, which threads they have opened and their depth setting. All state persists across sessions.

## 4. Adjustable Depth Mechanism

### 4.1 Three Discrete Levels

Labels describe explanation style, not user competence.

**Conceptual** (default for most users):

- Precise descriptions in plain language
- Analogies with explicit A-to-B mappings
- "Where this breaks down" note for every analogy
- Visuals show process flows, system diagrams, annotated step-throughs
- Prerequisites explained inline
- No formal notation except where it is simpler than the verbal equivalent (e.g., O(n))

**Technical**:

- Standard field terminology assumed
- Analogies available but collapsed by default
- Visuals add formal annotations (variable names, dimensions, function signatures)
- Key equations presented with verbal interpretation alongside
- Prerequisites flagged but not explained; linked to thread-capable definitions

**Formal**:

- Full mathematical treatment
- Proofs and derivations where relevant
- Visuals include computational graphs, probability graphical models
- Assumes graduate-level familiarity with the subfield

### 4.2 Five Coupled Dimensions

The single depth axis controls five dimensions simultaneously (independent per-dimension control deferred to post-MVP):

1. Assumed prerequisites (low: explained inline; high: assumed)
2. Vocabulary (low: jargon defined with analogies; high: standard terminology unscaffolded)
3. Granularity (low: more steps, more narration per step; high: compressed)
4. Context and motivation (low: more "why" before "what"; high: ratio shifts toward "what" and "how")
5. Mathematical formality (low: verbal and visual; high: formal notation)

The ordering follows an intuition-before-formalism principle: conceptual scaffolding dimensions escalate first, mathematical formality escalates last.

### 4.3 Calibration

During onboarding, the system presents a sample explanation at intermediate depth and lets the user adjust up or down. This outperforms self-assessment ("what's your level?") because users are poor self-assessors. A concrete example with "is this about right?" produces more accurate calibration.

### 4.4 Per-Section Inline Deepening

Each section has a control to go one level deeper on that section only. A user globally set to Conceptual who deepens one section sees that section at Technical while the rest stays Conceptual.

When the user deepens a section, visuals regenerate at the new depth. The transition between depth levels is animated so the user can track which new elements correspond to existing ones (temporal contiguity).

Deepening expands in two phases within a single response:

- **Phase 1**: Expand the intuitive explanation. More detail on why the mechanism exists, what problem it solves, what would go wrong without it. Refine or add the analogy. Update the visual to show more internal structure.
- **Phase 2**: Add a formal layer below. Introduce notation, equations or pseudocode. Provide a second visual (the formal version of the diagram). Include a "where this breaks down" note.

Phase 2 is visually distinct (different background, different visual style) so the user can identify and skip it.

### 4.5 Mismatch Detection (Signal Collection Only for MVP)

Track two signals per section:

- Time spent (unusually fast = too basic; unusually slow = too advanced)
- Thread-opening rate (frequent threads on basic concepts = depth too high)

On detected mismatch, surface a non-intrusive suggestion: "This section might work better at [other level]. Adjust?" Never auto-adjust. Collect signal data in V1; build an auto-adjustment model in V2.

## 5. Visual Explanation System

### 5.1 Content Type Classification

Each paper section is classified into one of eleven content types. Classification uses a two-stage pipeline:

**Stage 1 — Heuristic pre-classification**. Section title pattern matching, equation density, table presence, pseudocode detection, figure reference count and citation density. Accuracy estimate: 70-75% of sections correctly assigned.

**Stage 2 — LLM classification**. The LLM classifier receives the section text plus heuristic features and assigns a primary content type, an optional secondary type and the boundary between them (if applicable). The classifier also identifies sections where no visual adds value (introduction, conclusion, discussion) and assigns a NONE/SUMMARY label.

**Content types**:

|ID|Type|Typical trigger|
|---|---|---|
|1|Architecture Description|Model structure, component relationships|
|2|Algorithm / Process Walkthrough|Sequential procedures, pseudocode|
|3|Training / Optimization Procedure|Loss functions, gradient flow, update schedules|
|4|Data Pipeline / Preprocessing|Data transformation chains|
|5|Quantitative Results (Comparisons)|Benchmark tables, ablation studies|
|6|Quantitative Results (Distributions)|Embeddings, heatmaps, scaling curves|
|7|Qualitative Results / Case Studies|Input/output examples, attention maps|
|8|Theoretical Claim / Proof Structure|Theorems, bounds, proofs|
|9|Problem Formulation / Mathematical Setup|Variables, objectives, constraints|
|10|Conceptual Framework / Taxonomy|Categorization of methods or concepts|
|11|System / Deployment Architecture|Infrastructure, serving, communication|

### 5.2 Visual Format Selection

Each content type maps to a specific visual format at each depth level (full mapping in the Visual Explanation Taxonomy reference document). General patterns:

- **Conceptual**: Simplified block diagrams, flowcharts, annotated examples. Mermaid format where possible. 8-12 nodes max.
- **Technical**: Internal structure exposed, dimension annotations, parameter labels. Mermaid or SVG. 12-20 nodes.
- **Formal**: Computational graphs, full tensor operations, gradient flow. SVG required. 20-25 nodes max.

### 5.3 Visual Generation Pipeline

1. Classify the section (heuristic + LLM).
2. Select visual format from the content-type-to-format mapping table.
3. Generate the visual with a prompt specifying: content type, depth level, format, node limit and the constraint that every visual element must correspond to something in the text.
4. Render and verify (LLM self-check that the visual matches the text).

### 5.4 Design Constraints

All visuals follow these rules (derived from Mayer's principles):

- Positioned inline next to the text they explain (spatial contiguity)
- Text and visual appear simultaneously (temporal contiguity)
- Consistent color coding across an entire paper session: blue for input, green for processing, red for output (signaling). Legend included.
- No decorative elements; every diagram element corresponds to paper content (coherence)
- Short labels on diagrams, not full sentences duplicating the text (redundancy avoidance)
- Key term glossary (3-5 terms) before sections at Technical or Formal depth (pre-training)

## 6. Thread Model

### 6.1 Trigger

Always user-initiated. The user selects a concept, term, section or visual element and asks a question or requests more detail. The system can suggest threadable concepts via inline affordances on terms it detects as potentially unfamiliar, but never auto-opens a thread.

### 6.2 Context Inheritance

A thread inherits:

- The parent paper's full context
- The specific passage or visual that spawned the thread
- The user's current depth level
- Concept history from the main branch (the thread does not re-explain things the user has already seen)
- The user's demonstrated knowledge level

### 6.3 Context Return

When the user returns to the main branch:

- Terms explained in the thread lose their "unfamiliar" affordance in the main branch
- If the thread produced a useful analogy or diagram, a collapsed reference appears inline at the relevant point
- The thread remains accessible as a collapsible node

### 6.4 Thread Context Management

**MVP approach (summary injection)**:

- At branch creation: generate a 2-3 sentence summary of the main thread up to the branch point. Prepend to the branch thread's system prompt.
- At branch return: generate a 1-sentence summary of the branch discussion and append as a system note to the main thread.

Cost: one LLM call at branch creation, one at branch return.

### 6.5 MVP Constraints

- One level of thread depth. Threads cannot spawn sub-threads.
- If the user needs to go deeper within a thread, the thread itself expands.
- Thread visuals inherit the same color coding and signaling conventions as the main branch.
- Thread depth inherits the global setting by default. The user can adjust within a thread independently.

## 7. Session and Library

### 7.1 Library View

Every paper the user has touched is stored, grouped by status: Discovered, Evaluated, Reading, Completed. Each card shows the last interaction point for resuming.

### 7.2 Implicit Grouping

No explicit "research project" for MVP. Papers share a lineage if they originated from the same Discover query or were branched from the same Comprehend session. Display that lineage as a lightweight grouping without requiring the user to name or manage it.

### 7.3 Per-Paper Persisted State

- Evaluation summary
- Comprehension progress (sections expanded, threads opened)
- Depth setting at time of last interaction
- User bookmarks and highlights

### 7.4 Edge Cases

**Paper already in history**: Surface the existing evaluation with a note: "You evaluated this on [date]. Here's where you left off." Offer re-evaluate or resume.

**Discovery surfaces an already-evaluated paper**: Show it in results with a badge indicating prior evaluation and the user's verdict.

**Paper references another paper the user hasn't read**: In Comprehend, cited papers appear as interactive references. Clicking one triggers an inline micro-evaluation (3-sentence summary of what the cited paper contributes and why the current paper cites it) without leaving the comprehension view. From there the user can open a full Evaluate session as a branch.

## 8. Technical Architecture

### 8.1 External Dependencies

|Dependency|Purpose|Access|
|---|---|---|
|Semantic Scholar API|Paper search, metadata, citation graphs, TLDRs, SPECTER2 embeddings|Free at 1 req/sec; partner key at 100 req/sec|
|LLM API (e.g., Claude)|Explanation generation, visual generation, section classification, thread summarization, evaluation generation|Paid API access|
|PDF parser (Paper Mage or equivalent)|Parse uploaded PDFs into structured sections with figure/table/equation detection|Open-source|

### 8.2 Data Model (Core Entities)

```
User {
  id: string
  depth_calibration: "conceptual" | "technical" | "formal"
  onboarding_completed: boolean
}

Paper {
  id: string
  source: "semantic_scholar" | "user_upload" | "doi_lookup"
  semantic_scholar_id: string | null
  title: string
  authors: string[]
  year: number
  abstract: string
  sections: Section[]
  evaluation: Evaluation | null
  status: "discovered" | "evaluated" | "reading" | "completed"
}

Section {
  id: string
  paper_id: string
  title: string
  text: string
  content_type: ContentType
  secondary_content_type: ContentType | null
  order: number
}

Evaluation {
  id: string
  paper_id: string
  claim_summary: string
  method_overview: string
  method_visual: Visual
  evidence_assessment: string
  prerequisite_map: Prerequisite[]
  reading_time_estimate: { conceptual: number, technical: number, formal: number }
}

Conversation {
  id: string
  paper_id: string
  root_thread_id: string
}

Thread {
  id: string
  conversation_id: string
  parent_thread_id: string | null
  branch_point_message_id: string | null
  title: string
  status: "active" | "parked"
  messages: Message[]
}

Message {
  id: string
  thread_id: string
  role: "user" | "assistant"
  content: string
  paper_sections_referenced: string[]
  timestamp: datetime
}

Visual {
  id: string
  section_id: string | null
  content_type: ContentType
  depth_level: "conceptual" | "technical" | "formal"
  format: "mermaid" | "svg" | "html"
  content: string
}

ComprehensionState {
  id: string
  user_id: string
  paper_id: string
  depth_setting: "conceptual" | "technical" | "formal"
  per_section_depth: { section_id: string, depth: string }[]
  sections_expanded: string[]
  threads_opened: string[]
  bookmarks: Bookmark[]
  last_interaction: datetime
}
```

### 8.3 LLM Call Inventory

Each user action that requires an LLM call, for cost and latency estimation:

|Action|LLM calls|Notes|
|---|---|---|
|Evaluate a paper|1 (classification) + 1 (evaluation + visual)|Can be parallelized per section|
|Load a Comprehend section|1 (explanation + visual generation)|Per section, on demand|
|Deepen a section|1 (regenerate at new depth)||
|Open a thread|1 (summarize parent context)||
|Thread message|1 (response generation)||
|Return from thread|1 (summarize thread)||
|Discover query|0 (Semantic Scholar API only) + 1 (contribution summaries if TLDR unavailable)||
|Inline micro-evaluation of cited paper|1||

## 9. MVP Scope and Explicit Deferrals

### 9.1 In Scope

- Topic-based discovery via Semantic Scholar
- Paper evaluation from URL, DOI or PDF upload
- Section-by-section comprehension with visual explanations
- Three-level depth control (global + per-section)
- Single-level threading with summary-based context management
- Paper library with status tracking and implicit grouping
- Onboarding calibration flow
- Mismatch detection (signal collection only, no auto-adjustment)

### 9.2 Deferred to V2

- Cross-paper comparison (requires aligned section mapping)
- Independent per-dimension depth control
- Sub-threads (threads spawning threads)
- Auto-depth adjustment based on collected signals
- "Check your understanding" prompts (Gray & Holyoak principle 5)
- Audio walkthroughs (Mayer modality principle)
- Explicit research project management
- Non-CS paper support
